### T-023 | Изоляция очередей Sidekiq + батчевая обработка персон
**Категории:** ⚠️ Технический долг, 🎯 Архитектура, 👤 Пользовательский опыт
**Статус:** Активна
**Обоснование:** Один большой GED-файл порождает тысячи задач в одну очередь и блокирует обработку файлов других пользователей. Нужна очередная изоляция (разные очереди для разных типов воркеров) и батчинг — чтобы парсинг одного файла не монополизировал воркеры.
**Затрагивает:** Gedcom-сервис (требует доработки логики постановки задач в `app/workers/gedcom/`)
**Текущее состояние:** Все воркеры (`UploadWorker`, `TimelineWorker`, `PersonWorker` и т.д.) работают в одной дефолтной очереди `default`. Батчинга нет — каждая персона — отдельная задача, без лимита параллелизма.
**Подзадачи:**
- [ ] Завести отдельные очереди: `gedcom_parsing` (тяжёлый парсинг), `event_processing` (обработка событий), `timeline_build` (построение таймлайна), `default` (остальное)
- [ ] Назначить воркерам очереди через `sidekiq_options queue:`
- [ ] Настроить `config/sidekiq.yml`: приоритеты и `concurrency` для каждой очереди
- [ ] Добавить батчевую постановку задач для персон (например, `PersonWorker` ставится пачками по N через `bulk_perform_async` или аналог)
- [ ] Убедиться, что `GedcomFilesController` / `UploadWorker` не кладёт всё в одну очередь
