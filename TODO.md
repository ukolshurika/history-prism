### T-004 | Политика видимости событий
**Категории:** 🎯 Архитектура, 👤 Пользовательский опыт, 🔒 Безопасность

- [ ] Добавить `user_id` или `visibility` к событиям типа `person`
- [ ] Исправить `EventPolicy::Scope` — показывать личные события только владельцу
- [ ] Гарантировать, что таймлайны других пользователей не утекают личные факты
- [ ] Написать request-тесты на изоляцию данных

### T-022 | Доработка формы создания/редактирования событий — гибкий ввод дат
**Категории:** 👤 Пользовательский опыт, 👨‍💻 Разработка
**Статус:** Активна
**Обоснование:** Пользователь не может указать одну дату (без диапазона), приблизительную дату («около 1850») или только год. Форма не отражает реальные случаи ввода исторических данных. Блокирует качественное наполнение событий вручную.
**Текущее состояние:** Не реализовано. `FuzzyDate` на бэкенде поддерживает эти типы, но UI формы не предоставляет соответствующих полей ввода.
**Подзадачи:**
- [ ] Добавить в форму переключатель типа даты: одна дата / диапазон / только год / приблизительная
- [ ] Реализовать поле «около» (`ABT`) — один инпут + чекбокс/модификатор
- [ ] Реализовать поле «только год» — числовой инпут без месяца и дня
- [ ] Валидация на фронте: подсвечивать некорректные комбинации
- [ ] Убедиться, что бэкенд корректно принимает и сохраняет все типы через `FuzzyDate`
- [ ] **Баг:** `FuzzyDate` падает с ошибкой NOT UNIQUE при сохранении — нельзя указать одинаковую дату начала и окончания (например, однодневное событие)
- [ ] При клике на дату в таймлайне/списке — открывать форму редактирования этой даты напрямую, а после сохранения возвращать пользователя на страницу источника (откуда пришёл)

### T-023 | Изоляция очередей Sidekiq + батчевая обработка персон
**Категории:** ⚠️ Технический долг, 🎯 Архитектура, 👤 Пользовательский опыт
**Статус:** Активна
**Обоснование:** Один большой GED-файл порождает тысячи задач в одну очередь и блокирует обработку файлов других пользователей. Нужна очередная изоляция (разные очереди для разных типов воркеров) и батчинг — чтобы парсинг одного файла не монополизировал воркеры.
**Затрагивает:** Gedcom-сервис (требует доработки логики постановки задач в `app/workers/gedcom/`)
**Текущее состояние:** Все воркеры (`UploadWorker`, `TimelineWorker`, `PersonWorker` и т.д.) работают в одной дефолтной очереди `default`. Батчинга нет — каждая персона — отдельная задача, без лимита параллелизма.
**Подзадачи:**
- [ ] Завести отдельные очереди: `gedcom_parsing` (тяжёлый парсинг), `event_processing` (обработка событий), `timeline_build` (построение таймлайна), `default` (остальное)
- [ ] Назначить воркерам очереди через `sidekiq_options queue:`
- [ ] Настроить `config/sidekiq.yml`: приоритеты и `concurrency` для каждой очереди
- [ ] Добавить батчевую постановку задач для персон (например, `PersonWorker` ставится пачками по N через `bulk_perform_async` или аналог)
- [ ] Убедиться, что `GedcomFilesController` / `UploadWorker` не кладёт всё в одну очередь
